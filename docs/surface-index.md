# `surface_index.json` — loomcli surface enumeration

`surface_index.json` is a generated artifact at the repo root that
lists every "thing" loomcli exposes, with structured state flags.
Generated by `scripts/build_surface_index.py`. CI fails if the
committed file disagrees with what the emitter regenerates from the
current code.

This file is the **source of truth** for "what's in this CLI." It
exists because the 2026-05-09 demo failed when the `Workflow` kind
shipped in the JSON schema but had no handler in the CLI — and the
deploying agent misdiagnosed the resulting error as a runtime gap.
This index makes that class of drift impossible to ship by accident.

The companion repo (`powerloom-api`) has its own `surface_index.json`
with the same shape. Both are read by tracker threads, agent context
priming, and (eventually) automated cross-doc drift detection.

## When to regenerate

Whenever you change the CLI surface:
- Add/remove a `weave` subcommand
- Wire a new manifest kind in `KINDS` / `_APPLY_ORDER` / `_HANDLERS`
- Land or close a known-gap (a kind tracked in `surface_known_gaps.json`)
- Rename a CLI command

```
python scripts/build_surface_index.py
```

CI catches you if you forget — `pytest tests/test_surface_index_drift.py`
runs `--check` mode and fails on drift.

## Schema

```json
{
  "$schema_version": "1",
  "repo": "loomcli",
  "manifest_kinds": [...],
  "cli_commands": [...]
}
```

### `manifest_kinds[]`

One row per kind. The row exists if the kind appears in ANY of: the
JSON schema directory, `KINDS`, `_APPLY_ORDER`, `_HANDLERS`. The
fields tell you which registries it's in.

| Field | Type | Meaning |
|---|---|---|
| `name` | `str` | Canonical kind name (e.g. `"Workflow"`). |
| `schema_path` | `str \| null` | Path to `schema/v1/kinds/<kind>.schema.json` or null if no schema file. |
| `in_kinds_registry` | `bool` | True if `loomcli.manifest.schema.KINDS` contains this kind. |
| `handler_class` | `str \| null` | Fully-qualified handler class (`module:Class`) or null if no handler registered. |
| `apply_order` | `int \| null` | The kind's `_APPLY_ORDER` bucket, or null. |
| `category` | `str \| null` | Display category from `KindSpec.category` (`"identity"`, `"content"`, etc.). |
| `known_gaps` | `list` | Hand-annotated drift entries (see below). |

A "fully wired" kind has `in_kinds_registry=true`, `handler_class!=null`,
`apply_order!=null`, and `known_gaps==[]`. Anything else surfaces as a
drift signal.

### `cli_commands[]`

One row per leaf command and one row per command group, sorted by
path. Discovered via Typer introspection of `loomcli.cli:app`.

| Field | Type | Meaning |
|---|---|---|
| `path` | `str` | Full command invocation, e.g. `"weave thread create"`. |
| `module` | `str \| null` | Fully-qualified callback (`module:func`). Null for groups. |
| `summary` | `str \| null` | One-line description from `help=` or docstring. |
| `parent` | `str \| null` | Parent group path, or null for top-level. |
| `deprecated` | `bool` | True if the command is marked deprecated in Typer. |
| `is_group` | `bool` | True if this row is a sub-app (e.g. `weave thread`), false for leaves. |
| `known_gaps` | `list` | Hand-annotated drift entries. |

### `known_gaps[]` row shape

```json
{"kind": "missing_handler", "thread": "e1518478", "note": "..."}
```

| Field | Type | Meaning |
|---|---|---|
| `kind` | `str` | The drift category. Standard values: `"missing_handler"`, `"missing_apply_order"`, `"deprecated_pending_removal"`, `"parallel_path"`. |
| `thread` | `str` | Powerloom tracker thread ID (8-char prefix or full UUID) tracking the work to close the gap. **Required** — drift without a thread is just hidden drift. |
| `note` | `str \| null` | Human-readable rationale, optional. |

## Adding a known gap

When a kind or command is in a half-shipped state and you want CI to
stop failing while the proper fix is tracked elsewhere:

1. File a tracker thread describing the work to close the gap.
2. Add an entry to `surface_known_gaps.json` referencing that thread.
3. Regenerate: `python scripts/build_surface_index.py`.
4. Commit both files.

The drift test will pass because the gap is now expected. When the
underlying gap is fixed in code:

1. Remove the entry from `surface_known_gaps.json`.
2. Regenerate.
3. Mark the tracker thread done.

The conformance test (`test_kind_registry_conformance.py`) reads
`surface_index.json` and asserts every kind is either fully wired or
has matching `known_gaps` entries with thread references — same
enforcement as before, but the data lives in JSON instead of test
code.

## Why a JSON file (not a database, not a service)

- **Diffable**: `git log surface_index.json` answers "when did this
  surface change" with no extra tooling.
- **Reviewable**: Drift surfaces in PR diffs the same as code changes.
  Reviewers see "you renamed a command without updating its consumers."
- **Agent-readable**: Any LLM session can `cat surface_index.json` to
  ground its understanding of what the CLI exposes. No MCP call,
  no API request, no auth.
- **Reversible**: Delete the file + the script + the test if the
  pattern doesn't pan out. No migration, no deprecation arc.

Phase 2 of the parent arc (Powerloom thread `c77b8922`) considers
extending this with structured tracker-thread surfaces, but that's
deferred until this artifact has soaked in real use.
